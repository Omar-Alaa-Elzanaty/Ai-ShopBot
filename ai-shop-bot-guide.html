02 — CORE QUESTIONS ANSWERED
Your 4 Key Problems — Solved
Q1

How will you detect which method the user needs to execute and pass the right parameters?

This is exactly what Microsoft Semantic Kernel's Function Calling (also called Auto Function Invocation) was built for. You don't write your own intent classifier. Instead, here's the mental model:

You define your business methods (Search, CreateOrder, AddToCart, etc.) as SK Plugins. Each plugin function has a description written in plain English like "Creates an order for a product given its ID and quantity". You also annotate each parameter with a description like "The unique MongoDB product ID".

When the user sends a message, you pass it to the LLM (GPT-4o) along with all your registered plugins. The model reads the conversation history + user message, then decides on its own which function to call and extracts the parameters from the conversation. It returns a structured JSON tool call like CreateOrder(productId: "abc123", qty: 1). Your SK kernel intercepts that, invokes your real .NET method, and returns the result back to the model to formulate a natural response.

Key insight: You never write an if/else intent switch. The LLM is the router. Your job is to write clear plugin descriptions so the model makes the right routing decision.
Q2

Do you need two different AI models — one for chat and one for vector search (embeddings/RAG)?

Yes, you need two different model types — but from the same provider. They serve completely different purposes and cannot be swapped.

Chat / Reasoning Model (e.g. gpt-4o or gpt-4o-mini): This is the brain. It reads conversation history, understands intent, decides which plugin to call, and generates the natural language responses the user sees. It cannot create embeddings.

Embedding Model (e.g. text-embedding-3-small): This is a mathematical transformer. It converts text into a dense numeric vector (array of floats). You use it in two places: (1) at product ingestion time — embed each product description and store it in MongoDB, (2) at query time — embed the user's search phrase and run $vectorSearch in MongoDB to find semantically similar products. It has no conversational ability at all.

In practice: Semantic Kernel manages both. You register both models with the kernel. Your SearchProducts plugin internally calls the embedding model, does the MongoDB vector search, and returns results. The chat model never knows this happened — it just sees the returned product list.
Q3

Do you need a design pattern like Factory Pattern to handle method selection?

No — the Factory Pattern is not needed here, and using it would be a mistake. The Factory Pattern solves the problem of "I need to decide at runtime which concrete class to instantiate." But in your case, the LLM is already making that decision through Function Calling.

If you try to build a Factory on top of LLM function calling, you end up with double routing logic: the LLM picks a function, then your factory also tries to pick an implementation. This creates unnecessary complexity and two sources of truth for routing.

The pattern that does apply naturally here is the Plugin Pattern (which is Semantic Kernel's own term). Think of each plugin class as a self-contained capability module: ProductPlugin, OrderPlugin, CartPlugin, ReviewPlugin. Each one registers its functions with SK. This is structurally similar to a Strategy Pattern — each plugin encapsulates a behavior — but without a separate factory to instantiate them.

One genuine use-case for a pattern: If you want multiple search strategies (vector search vs. keyword fallback vs. category filter), a Strategy Pattern inside your ProductPlugin makes sense. But that's an internal implementation detail, not the top-level routing.
Q4

In Clean Architecture, should every operation be CQRS (Commands/Queries) or can you use a single service class?

CQRS is the right call here, and here's why it matters specifically for your project — not just as a rule.

Your system has a clear read/write split already baked in by design. Reads (SearchProducts, GetOrderHistory, GetCart) hit MongoDB + Redis and are called frequently in chat context. Writes (CreateOrder, AddToCart, SubmitReview) hit MSSQL and have side effects like sending notifications or updating inventory. If you put all of this in a single ShopService class, it will grow to hundreds of lines and become impossible to test in isolation.

With CQRS + MediatR: each SK Plugin method simply creates a Command or Query object and dispatches it via mediator.Send(). The handler lives in the Application layer and knows nothing about SignalR or Semantic Kernel. This means you can unit test every business operation independently, add pipeline behaviors (validation, logging, caching) in one place, and your SK plugins stay thin — they're just translators between the AI layer and your application core.

Practical rule: Any operation that changes state = Command. Any operation that only reads data = Query. Your SK plugin methods become one-liner dispatchers. All business logic lives in handlers.
03 — REQUEST LIFECYCLE
How a User Message Gets Processed
1

User sends message via SignalR Hub

Client sends "I need a summer jacket around 200 EGP" over WebSocket connection. The Hub authenticates the user via JWT, retrieves the session ID, and passes the message to the Chat Orchestrator.

2

Load conversation history from Redis

The orchestrator fetches the last N messages for this session from Redis. This gives the LLM full context — the user may have said "the blue one" referring to a product mentioned 2 turns ago. Redis TTL handles session expiry automatically.

3

Semantic Kernel sends to LLM with plugins registered

SK builds the full prompt: system instructions + conversation history + current message + list of available plugin functions with their descriptions. This is sent to GPT-4o. The model does not execute anything — it responds with a tool call decision.

4

LLM returns: call SearchProducts("summer jacket", maxPrice: 200)

The model determines the user wants to search, not order. It returns a structured function call with extracted parameters. SK automatically intercepts this before it reaches your code — this is Auto Function Invocation.

5

SK invokes your SearchProducts plugin method

Your plugin method fires. Internally it: (a) calls the embedding model to vectorize "summer jacket", (b) runs $vectorSearch in MongoDB with a price filter, (c) returns top 5 matching products as a JSON list back to SK.

6

LLM generates final natural language response

SK passes the function result back to the model. The model now formats a friendly response: "I found 3 summer jackets in your budget! Here's what caught my eye..." along with the structured product data.

7

Save to Redis, stream back via SignalR

Both the user message and assistant response are appended to Redis history. The response is streamed back to the client in real-time through the SignalR Hub — the user sees tokens appearing as they're generated.

04 — TECHNOLOGY STACK
Every Tool and Its Exact Role
🧠
AI FRAMEWORK

Microsoft Semantic Kernel

Orchestrates the LLM, manages plugins (your methods), handles Auto Function Invocation, and chains multi-step AI operations. The core brain of routing.

🔍
EMBEDDINGS + SEARCH

MongoDB Atlas Vector Search

Stores product embeddings. Runs $vectorSearch aggregation pipeline for semantic product matching. Also stores product catalog (flexible schema).

⚡
REAL-TIME

SignalR (ASP.NET Core)

Persistent WebSocket connection for chat. Enables streaming responses token-by-token. Handles connection groups per user session.

🗃️
TRANSACTIONAL DATA

MSSQL + EF Core

Stores orders, cart items, user profiles. ACID transactions for order creation and payment. Strong consistency where it matters.

🔐
AUTHENTICATION

ASP.NET Identity + JWT

User registration/login. JWT for stateless API auth. Refresh token rotation. SignalR hub authentication via Bearer token in query string.

⚙️
CACHE + HISTORY

Redis (StackExchange.Redis)

Stores serialized conversation history per session with TTL. Also cacheable: product results, user cart summary. Prevents redundant DB calls mid-conversation.

📐
ARCHITECTURE

Clean Architecture + CQRS

Domain → Application (MediatR handlers) → Infrastructure → API. SK Plugins live in Application layer as thin dispatchers to MediatR commands/queries.

🔢
EMBEDDING MODEL

text-embedding-3-small (OpenAI)

Converts product descriptions and user queries into 1536-dimension vectors. Called during product ingestion pipeline and at search time. Very cheap per token.

05 — DESIGN DECISIONS
Pattern Usage Summary
Pattern / Approach Use It? Where & Why
SK Plugin Pattern ✓ YES Your primary method-routing mechanism. Each plugin class groups related capabilities (ProductPlugin, OrderPlugin, CartPlugin, ReviewPlugin). This IS your architecture for intent → action.
CQRS + MediatR ✓ YES Inside Application layer. Every SK plugin method dispatches a Command or Query. Keeps plugins thin, handlers testable, and adds pipeline behavior support (validation, logging, caching).
Repository Pattern ✓ YES In Infrastructure layer. IProductRepository, IOrderRepository. Your MediatR handlers depend on these interfaces, not concrete DB implementations. Makes testing clean.
Strategy Pattern ⚡ OPTIONAL Useful inside ProductPlugin if you want fallback search logic: try vector search → if low confidence, fallback to keyword search. Inject as ISearchStrategy.
Factory Pattern ✗ NO Not needed. The LLM + SK already handles plugin/method selection. Adding a factory creates redundant routing logic and two sources of truth.
Observer / Event-Driven ⚡ LATER Useful when orders trigger notifications or inventory updates. MediatR's INotification handles this well — publish an OrderCreatedEvent after writing to DB.
Singleton for SK Kernel ✓ YES Register the Semantic Kernel instance as a singleton in DI container. It's thread-safe and expensive to create. Plugins can be transient/scoped if they depend on user context.
06 — CLEAN ARCHITECTURE MAP
What Lives Where
DOMAIN LAYER

No dependencies on anything external.

  • Product, Order, Cart, Review entities
  • Value Objects (Money, ProductId)
  • Domain Events (OrderCreated)
  • Repository interfaces (IOrderRepository)
APPLICATION LAYER

Business logic. References only Domain.

  • MediatR Commands & Queries
  • Command/Query Handlers
  • SK Plugin classes (thin dispatchers)
  • DTOs and mapping logic
INFRASTRUCTURE LAYER

All external dependencies.

  • MongoDB + EF Core implementations
  • Redis chat history service
  • OpenAI embedding service
  • ASP.NET Identity configuration
API / PRESENTATION LAYER

Entry points only. No business logic.

  • SignalR Hub (ChatHub.cs)
  • REST Controllers (Auth, Admin)
  • SK Kernel registration + DI setup
  • Middleware (JWT, error handling)
